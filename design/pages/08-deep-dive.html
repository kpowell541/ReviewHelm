What It Means
When goroutines are spawned, they should check for context cancellation to avoid leaking resources when the parent context is done.
Why It Matters
Leaked goroutines hold open connections, consume memory, and can cause cascading failures in production services.
How to Verify
Check for select blocks that listen on ctx.Done() alongside channel operations.
Code Examples
❌ Bad
go func() { result := doWork() ch <- result }()
✅ Good
go func() { select { case <-ctx.Done(): return case ch <- doWork(): } }()
Key Takeaway
Every goroutine that outlives its caller should respect context cancellation to prevent resource leaks.

Can you explain why context cancellation matters for goroutines?
Great question! When a goroutine doesn't check for context cancellation, it continues running even after its parent has moved on. This is called a "goroutine leak." Think of it like leaving a faucet running — each leaked goroutine holds onto resources (memory, connections, CPU) indefinitely.
How do I spot this in a code review?
Look for these patterns: 1. Any `go func()` that does blocking work without a `select` on `ctx.Done()` 2. Loops inside goroutines that don't check cancellation 3. Channel sends without a context-aware timeout The key tell is: "If this goroutine's caller returns early, will this goroutine notice?"
